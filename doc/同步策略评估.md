# 同步策略评估

> 基于《数据持久化方案》和《数据库表设计》中的同步设计，结合当前代码现状（纯浏览器原型 + localStorage，无 SQLite / 无同步代码）进行评估。

---

## 一、设计摘要

| 机制 | 设计内容 |
|------|---------|
| 增量同步 | `sync_changelog` 表记录每次 deal/image 变更，按设备分文件上传（`changelog_{deviceId}.jsonl`） |
| 全量同步 | 手动触发，上传/下载完整 db + 所有图片，覆盖目标端，清空 changelog |
| 状态追踪 | `sync_meta` 单行表记录 `localRevision` / `remoteRevision`，push/pull 时对比 |
| 冲突处理 | LWW（Last-Write-Wins），`updated_at` 较新的一方胜出；删除进回收站，需对方确认 |
| 自动同步 | 增量 only：本地变更 debounce 3s 后自动 push，前台恢复时自动 pull |
| 传输层 | WebDAV PUT，远端目录结构 `full/` + `changelog_{deviceId}.jsonl` + `images/` + `snapshots/` |

---

## 二、✅ 可行的部分

### 2.1 增量同步 + revision 递增

设计合理。用 `sync_changelog` 表追踪变更，`sync_meta` 记录两端 revision，只传输增量。JSONL 格式适合顺序追加，避免"整个数据库上传"的浪费。

**前提条件**：必须先实现 SQLite 层，目前连数据库都没有。

### 2.2 Push debounce 3 秒

对于手动保存场景合理，避免频繁触发。但当前原型已实现 Vue `watch(deep:true)` 自动保存，如果 debounce 在 Repository 层而非 watch 层，changelog 写入频率仍然很高。

### 2.3 设备标识 + `sync_meta` 单行状态表

设计简洁。每台设备一个 UUID，记录 `localRevision` / `remoteRevision`，对于个人工具类 app 足够。

---

## 三、⚠️ 有风险的部分

### 3.1 LWW 对删除操作不友好 → 数据丢失

**文档设计**：一边删除一边修改 → 优先保留删除。

**风险场景**：

1. 手机 A 在离线状态下编辑了一条优惠，保存（`updated_at` 更新）
2. 手机 B 同时删除了这条优惠（同步后 `deleted=1`）
3. 下次同步 → A 的修改被丢弃，优惠消失

优惠跟踪 app 中，用户精心编辑的内容被静默删除是糟糕体验。文档提到"可选弹窗让用户选择"，但 LWW 和用户选择是两种不同策略，文档混用了，实现时需要明确走哪条路。

**建议**：删除操作不直接写 `deleted=1`，而是先进"回收站"（软删除 + 标记 `pending_delete`），同步时对方看到 `pending_delete` 再弹窗确认。

### 3.2 changelog.jsonl 追加写入会无限膨胀

**文档设计**：push 时将本地 changelog 序列化为 JSONL → 追加写入远端 `changelog.jsonl`。

**问题**：

- 每条优惠的每次编辑/删除都会写一行，一年后可能有数千行
- pull 时每次都要下载完整文件再按 revision 过滤，浪费带宽
- WebDAV 不支持"只读取文件末尾"，必须全量下载

**建议**：

- 方案 A：changelog 按 revision 范围分文件（如 `changelog_0001-1000.jsonl`）
- 方案 B：定期压缩——远端保留最新 `changelog.jsonl` + 最新 `snapshots/*.db.zip`，pull 时只下载 `remoteMeta.json` 对比 revision，按需拉取

### 3.3 image 和 deal 同步的时序问题

**文档设计**：deal 变更和图片变更分别写 `sync_changelog`，push 流程是先上传 changelog.jsonl 再上传变更图片。

**问题**：步骤 1 成功但步骤 2 失败（网络中断）时，对方 pull 到了 deal 数据但图片缺失。文档没有描述这个中间状态的处理。

**建议**：

- 方案 A：图片上传先于 changelog 写入，保证 deal 数据到达时图片已在
- 方案 B：在 deal 记录中用 `image_status` 字段标记（`synced` / `pending`），pull 端遇到 `pending` 时显示占位图

### 3.4 并发 push 的竞态条件

**文档设计**：读取远端 `remoteMeta.json` → 对比 revision → 有更新则先 pull → 追加 changelog → 上传。

**问题**：两台设备同时 push 时：

1. 设备 A 读 `remoteMeta`（revision=100）
2. 设备 B 读 `remoteMeta`（revision=100）
3. 设备 A 写入 changelog（revision=101-105）
4. 设备 B 写入 changelog（revision=101-103），**覆盖了 A 的 104-105**

WebDAV PUT 是全量覆盖，不支持原子追加。

**建议**：

- 方案 A（乐观锁）：用 `ETag` 或 `If-Match` 头，冲突时重试
- 方案 B（设备隔离）：changelog 按设备分文件（`changelog_{deviceId}.jsonl`），pull 时合并所有设备文件，天然避免冲突 ← **推荐**

### 3.5 自动拉取 30 分钟 + 定时器休眠

**文档设计**：每 30 分钟自动 pull 一次。

**问题**：在手机端（PWA / Capacitor），浏览器标签页被后台挂起时 `setInterval` 会被节流或暂停，30 分钟间隔在移动端基本不可靠。

**建议**：pull 改为"页面可见时立即拉取一次 + 监听 `visibilitychange` 事件"，而不是固定定时器。

## 四、🔴 不可行 / 需要重新设计的部分

### 4.1 SQLite 在纯浏览器环境不可用

当前原型是纯浏览器 app（CDN 加载 Vue 3，无构建工具），而文档选择的是 SQLite 3（C 库）。可行的浏览器替代方案：

| 方案 | 特点 | 与文档 DDL 兼容性 |
|------|------|------------------|
| **sql.js** | SQLite 编译为 WASM，内存中运行，需手动持久化到 IndexedDB | ✅ 完全兼容 |
| **wa-sqlite** | 更现代的 WASM SQLite，支持 OPFS 持久化（Chrome 102+） | ✅ 完全兼容 |
| **absurd-sql** | sql.js + IndexedDB 后端，透明持久化，但维护不活跃 | ✅ 完全兼容 |
| **Dexie.js** | IndexedDB 包装器，非 SQL 但 API 友好，天然支持浏览器 | ❌ 需重写查询层 |

文档的 DDL 和 Repository 设计假设了原生 SQLite SQL 语法。如果用 sql.js / wa-sqlite 可以基本复用；如果用 Dexie 则需要重写所有查询。

### 4.2 WebDAV PUT 不支持"追加写入"

文档说"追加写入远端 `changelog.jsonl`"，但 WebDAV 协议只有 PUT（全量覆盖），没有 APPEND。实现时必须：

1. 先 GET 下载完整文件
2. 在内存中拼接新行
3. 再 PUT 上传完整文件

这在并发场景下有覆盖风险（见 3.4），且每次 push 的流量是 `旧内容 + 新内容`，不是只有增量。

### 4.3 zip 备份/恢复需要引入 JSZip

当前没有任何构建工具和 npm 依赖。实现 zip 功能需要引入 JSZip（~100KB gzip），这意味着：

- 要么引入构建工具（Vite / Webpack）
- 要么 CDN 加载 JSZip

## 六、结论

同步策略的**大方向**（增量 changelog + revision + LWW）是成熟方案，适合个人工具类 app 的"多设备互通"场景。

原始设计在 WebDAV 协议限制上有几处理想化假设，经评估后已确认以下修订：

| 原始设计 | 修订后 |
|----------|--------|
| 只有增量同步 | 新增全量 Push/Download（手动触发），远端增加 `full/` 目录 |
| 所有设备追加同一 `changelog.jsonl` | 按设备分文件 `changelog_{deviceId}.jsonl`，天然避免并发覆盖 |
| 删除直接写 `deleted=1` | 回收站机制：`deleted=2`（待确认）→ 对方弹窗 → `deleted=1` 或 `deleted=0` |
| 先写 changelog 再上传图片 | 先上传图片，成功后写 changelog，保证数据一致性 |
| 定时 30min pull | App 前台恢复时立即 pull（`visibilitychange`），定时为辅 |

> 注：不可行部分（SQLite WASM、WebDAV 无追加等）仅针对纯浏览器环境，当前产品为移动端 App，可使用原生 SQLite 和平台级网络 API，不适用。
