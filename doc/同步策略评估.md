# 同步策略评估

> 基于《数据持久化方案》和《数据库表设计》中的同步设计，结合当前代码实现状态进行评估。
> 
> **最后更新**：2026-06-13 | **实现状态**：增量同步核心逻辑已全部实现并编译通过

---

## 一、设计摘要

| 机制 | 设计内容 |
|------|---------|
| 增量同步 | `sync_changelog` 表记录每次 deal/image 变更，按设备分文件上传（`changelog_{deviceId}.jsonl`） |
| 全量同步 | 手动触发，上传/下载完整 db + 所有图片，覆盖目标端，清空 changelog |
| 状态追踪 | `sync_meta` 单行表记录 `localRevision` / `remoteRevision`，push/pull 时对比 |
| 冲突处理 | LWW（Last-Write-Wins），`updated_at` 较新的一方胜出；删除进回收站，需对方确认 |
| 自动同步 | 增量 only：本地变更 debounce 3s 后自动 push，前台恢复时自动 pull |
| 传输层 | WebDAV PUT，远端目录结构 `full/` + `changelog_{deviceId}.jsonl` + `images/` + `snapshots/` |

## 二、✅ 已实现的部分

### 2.1 增量同步 + revision 递增

设计合理。用 `sync_changelog` 表追踪变更，`sync_meta` 记录两端 revision，只传输增量。

**实现状态**：已完成。`DealDao.saveDeal` / `softDeleteDeal` / `hardDeleteDeal` / `restoreDeal` 均在事务外调用 `_logChange`，通过 `SyncDao.nextRevision()` 原子递增版本号并写入 `sync_changelog`。

### 2.2 Push debounce 3 秒

对于手动保存场景合理，避免频繁触发。

**实现状态**：当前由 UI 层手动触发同步（`smartSync`），未实现自动 debounce。后续可在 Riverpod Provider 层添加 `Timer` 实现保存后 3s 自动 push。

### 2.3 设备标识 + `sync_meta` 单行状态表

设计简洁。每台设备一个 UUID，记录 `localRevision` / `remoteRevision`，对于个人工具类 app 足够。

**实现状态**：已完成。`SyncService.getDeviceId()` 使用 `uuid` 包生成 v4 UUID，首次生成后持久化到 `sync_meta.device_id`。

## 三、⚠️ 有风险的部分

### 3.1 LWW 对删除操作不友好 → 数据丢失

**文档设计**：一边删除一边修改 → 优先保留删除。

**风险场景**：

1. 手机 A 在离线状态下编辑了一条优惠，保存（`updated_at` 更新）
2. 手机 B 同时删除了这条优惠（同步后 `deleted=1`）
3. 下次同步 → A 的修改被丢弃，优惠消失

优惠跟踪 app 中，用户精心编辑的内容被静默删除是糟糕体验。

**当前实现**：`softDeleteDeal` 写入 `operation='pending_delete'`，但 `_applyChanges` 中直接执行 `softDeleteDeal`，**尚未实现弹窗确认机制**。后续需在 Pull 到 `pending_delete` 时弹窗让用户选择保留或删除。

### 3.2 changelog.jsonl 追加写入会无限膨胀

**文档设计**：push 时将本地 changelog 序列化为 JSONL → 追加写入远端 `changelog.jsonl`。

**实际实现**：已改为 **PUT 覆盖写入**（非追加）。每次 Push 时：
1. 读取本地 `synced_at IS NULL` 的待同步记录
2. 序列化为 JSONL（upsert 记录附加完整 deal payload）
3. PUT 覆盖上传 `changelog_{deviceId}.jsonl`
4. 成功后调用 `purgeSyncedChanges()` **删除本地已同步记录**

因此本地 `sync_changelog` 表和远端 jsonl 文件均**不会无限增长**。

### 3.3 image 和 deal 同步的时序问题

**文档设计**：deal 变更和图片变更分别写 `sync_changelog`，push 流程是先上传 changelog.jsonl 再上传变更图片。

**实际实现**：已改为 **先上传图片，再上传 changelog**。`incrementalPush` 流程：
1. 遍历待同步的 deal upsert 记录
2. 查询关联 `dealImages`，将图片文件上传至 `images/{dealId}.jpg`
3. 图片上传成功后，再序列化并上传 changelog jsonl

Pull 时 `_applyChanges` 会下载远端图片到本地 `images/` 目录，保证数据一致性。

### 3.4 并发 push 的竞态条件

**文档设计**：读取远端 `remoteMeta.json` → 对比 revision → 有更新则先 pull → 追加 changelog → 上传。

**实际实现**：已采用 **设备隔离方案**。每台设备的 changelog 独立存储为 `changelog_{deviceId}.jsonl`，`incrementalPull` 时合并所有设备文件并按 `entityType|entityId` 取最大 revision（LWW），天然避免并发覆盖。

### 3.5 自动拉取 30 分钟 + 定时器休眠

**文档设计**：每 30 分钟自动 pull 一次。

**实际实现**：当前为**纯手动触发**（用户点击"立即同步"）。未实现自动定时拉取。后续可通过 `WidgetsBindingObserver.didChangeAppLifecycleState` 监听 App 恢复前台时自动触发 `smartSync`。

## 四、🔴 不可行 / 已解决的部分

### 4.1 SQLite 在纯浏览器环境不可用

> **已解决**：当前产品为 Flutter 移动端 App，使用 `drift` 库 + 原生 SQLite，无需浏览器 WASM 方案。

### 4.2 WebDAV PUT 不支持"追加写入"

> **已解决**：实现改为 PUT 覆盖写入 `changelog_{deviceId}.jsonl`（每次只包含当前未同步记录），配合本地 `purgeSyncedChanges()` 清理已同步记录，不依赖追加语义。

### 4.3 zip 备份/恢复需要引入 JSZip

> **已解决**：使用 Flutter `archive` 包处理 zip，备份服务 `BackupService` 已实现完整的 export/import 逻辑（含数据库 + images 打包）。

## 六、结论

同步策略的**大方向**（增量 changelog + revision + LWW）是成熟方案，适合个人工具类 app 的"多设备互通"场景。

当前实现的核心代码位于 `lib/core/sync/sync_service.dart`，完整闭环如下：

### 6.1 增量同步完整闭环

```mermaid
flowchart TD
    A[用户保存/删除 Deal] --> B{DealDao}
    B -->|saveDeal| C[_logChange operation=upsert]
    B -->|softDelete| D[_logChange operation=pending_delete]
    B -->|hardDelete| E[_logChange operation=delete]
    C --> F[sync_changelog<br/>revision++, synced_at=NULL]
    D --> F
    E --> F
    F --> G[用户触发 smartSync]
    G --> H[Pull 增量]
    H --> H1[扫描远端 changelog_*.jsonl]
    H1 --> H2[下载各设备 changelog]
    H2 --> H3[LWW 合并<br/>entityType|entityId 取最大 revision]
    H3 --> H4[返回 List&lt;SyncChangeEntry&gt;]
    H4 --> H5[_applyChanges]
    H5 --> H5a[upsert → DealDao.saveDeal<br/>图片从远端下载到本地]
    H5 --> H5b[pending_delete → softDeleteDeal]
    H5 --> H5c[delete → hardDeleteDeal]
    H5a --> I[Push 增量]
    H5b --> I
    H5c --> I
    I --> I1[遍历待同步记录]
    I1 --> I2[上传关联图片<br/>images/{dealId}.jpg]
    I2 --> I3[序列化 JSONL<br/>upsert 附加 deal payload]
    I3 --> I4[PUT 覆盖<br/>changelog_{deviceId}.jsonl]
    I4 --> I5[标记 synced_at=now]
    I5 --> I6[purgeSyncedChanges<br/>删除已同步记录]
    I6 --> J[结束]

    K[用户触发 fullUpload] --> L[exportBackup<br/>db + images → zip]
    L --> M[上传 zip → full/backup.zip]
    M --> N[上传 images/*.jpg]
    N --> O[_cleanupOrphanRemoteImages<br/>删除远端废弃图片]
    O --> J
```

### 6.2 Push 增量详细流程

```mermaid
sequenceDiagram
    participant A as 本地数据库
    participant B as SyncService
    participant C as 传输层
    participant D as 远端存储

    B->>A: getPendingChanges(deviceId)<br/>WHERE synced_at IS NULL
    A-->>B: List&lt;SyncChangelogData&gt;
    loop 每条 upsert 记录
        B->>A: SELECT dealImages WHERE dealId=? AND deleted=0
        A-->>B: DealImage
        B->>C: upload images/{dealId}.jpg
        C->>D: PUT images/{dealId}.jpg
    end
    B->>B: 序列化 JSONL（含 deal payload）
    B->>C: upload changelog_{deviceId}.jsonl
    C->>D: PUT 覆盖 changelog_{deviceId}.jsonl
    B->>A: markSynced(id) + updateRevision()
    B->>A: purgeSyncedChanges()<br/>DELETE WHERE synced_at IS NOT NULL
```

### 6.3 Pull 增量详细流程

```mermaid
sequenceDiagram
    participant D as 远端存储
    participant C as 传输层
    participant B as SyncService
    participant A as 本地数据库

    B->>C: list 远端 changelog_*.jsonl
    C->>D: PROPFIND /zheduoduo/
    D-->>C: 文件列表
    C-->>B: List&lt;RemoteFileInfo&gt;
    loop 每个设备 changelog
        B->>C: download changelog_{deviceId}.jsonl
        C->>D: GET changelog_{deviceId}.jsonl
        D-->>C: JSONL bytes
        C-->>B: Uint8List
    end
    B->>B: 解析 JSONL → 按 entityType|entityId 去重<br/>保留最大 revision（LWW）
    B-->>B: 返回 List&lt;SyncChangeEntry&gt;
    loop 每条变更
        alt upsert
            B->>C: download images/{dealId}.jpg
            C->>D: GET images/{dealId}.jpg
            D-->>C: image bytes
            C-->>B: Uint8List
            B->>A: saveDeal(DealWithDetails)<br/>写入 db + 保存图片到 images/
        else pending_delete
            B->>A: softDeleteDeal(dealId)
        else delete
            B->>A: hardDeleteDeal(dealId)
        end
    end
```

| 原始设计 | 实际实现 |
|----------|--------|
| 只有增量同步 | 增量（`smartSync`）+ 全量（`fullUpload` / `fullDownload`） |
| 所有设备追加同一 `changelog.jsonl` | 按设备分文件 `changelog_{deviceId}.jsonl`，PUT 覆盖 |
| 删除直接写 `deleted=1` | `deleted=2`（`pending_delete`）待确认机制 |
| 先写 changelog 再上传图片 | **先上传图片，再上传 changelog** |
| 定时 30min pull | **纯手动触发**，生命周期驱动待实现 |
| changelog 无限增长 | **每次 Push 后 purge 已同步记录**，不会增长 |
