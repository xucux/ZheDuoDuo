# 增量同步方案可行性评估

## 一、背景与目标

本文评估一套新的增量同步方案，并与折多多（ZheDuoDuo）现有同步机制进行对比，最终给出可行性结论与实施建议。

---

## 二、现有同步方案分析

### 2.1 核心架构

现有方案位于 `lib/core/sync/sync_service.dart`，采用 **"变更日志 + JSONL 增量文件 + 全量 ZIP 备份"** 的混合策略：

| 组件 | 说明 |
|------|------|
| `sync_changelog` 表 | 记录每次数据变更的元信息 |
| `sync_meta` 表 | 存储设备 ID 与版本号（revision） |
| 增量推送 | 读取本地 pending changelog → 组装完整 deal payload → 上传为 `changelog_{deviceId}.jsonl` |
| 增量拉取 | 下载所有设备的 JSONL → 按 `entityType+entityId` 去重保留最新 revision（LWW） → 应用变更 |
| 全量上传 | 导出 ZIP（SQLite + images + manifest）→ 上传到 `full/` 目录 |
| 全量下载 | 下载 ZIP → 关闭数据库 → 替换文件 → 重启应用 |

### 2.2 现有 `sync_changelog` 表结构

```sql
CREATE TABLE sync_changelog (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  device_id TEXT NOT NULL,
  entity_type TEXT NOT NULL,    -- 实体类型（如 'deal'）
  entity_id TEXT NOT NULL,      -- 实体主键
  operation TEXT NOT NULL,      -- 操作类型（upsert / pending_delete / delete）
  revision INTEGER NOT NULL,    -- 单调递增版本号
  changed_at DATETIME NOT NULL,
  synced_at DATETIME,           -- NULL 表示待同步
  payload_hash TEXT             -- 预留字段，目前未使用
);
```

### 2.3 现有方案触发点

变更日志的写入集中在 `DealDao` 中：

- `saveDeal()` → `_logChange(id, 'upsert')`
- `softDeleteDeal()` → `_logChange(id, 'pending_delete')`
- `hardDeleteDeal()` → `_logChange(id, 'delete')`
- `restoreDeal()` → `_logChange(id, 'upsert')`

**关键缺陷**：`_logChange` 仅在 `DealDao` 中调用，**不记录** `app_settings`、`ai_configs`、`secrets`、`prompts`、`image_compress_settings`、`backup_records` 等表的变更。这意味着用户在 AI 配置、应用设置等页面的修改不会被同步到其他设备。

### 2.4 现有方案的其他问题

1. **无全局总开关**：是否记录 changelog 依赖 `DealDao` 构造时是否传入 `_syncDao`，而非用户可控制的云同步总开关。
2. **全量上传后未清空 changelog**：全量备份上传后，本地旧的增量记录仍保留，可能导致下次增量推送时重复上传已包含在全量备份中的数据。
3. **全量下载后未处理本地 changelog**：直接覆盖数据库，本地未同步的变更会丢失，且无提示。
4. **附件未独立标记**：图片随 deal payload 一起处理，没有显式的"附件清单"概念，难以支持"仅同步变更附件"的优化。

## 三、新方案要点梳理

### 3.1 用户提出的六项要求

| 序号 | 要求 | 说明 |
|------|------|------|
| 1 | 所有表需有主键 | 作为变更记录的唯一标识 |
| 2 | 云同步总开关 | 仅开启后才记录增量/变化量 |
| 3 | 新 changelog 表 | 记录表名、变更时间、主键、方式、SQL 语句、是否涉及附件、附件地址 |
| 4 | 全量上传后清空 changelog | 表示其他设备以本次全量为基线 |
| 5 | 全量下载后弹窗确认 | 本地 changelog 存在时，让用户选择"清空"或"合并" |
| 6 | 推送时克隆 changelog | 将 changelog 数据和变更附件一起推送到远端 `changelog/` 目录 |

## 四、数据库主键检查

对项目内全部 13 张表进行主键审查：

| 表名 | 主键字段 | 主键类型 | 状态 |
|------|---------|---------|------|
| `deals` | `id` (TEXT) | 单字段主键 | ✅ |
| `deal_tags` | `dealId` + `tag` | 联合主键 | ✅ |
| `deal_promotions` | `dealId` + `sortOrder` | 联合主键 | ✅ |
| `coupons` | `id` (INTEGER AUTOINCREMENT) | 单字段主键 | ✅ |
| `deal_images` | `dealId` (TEXT) | 单字段主键 | ✅ |
| `app_settings` | `key` (TEXT) | 单字段主键 | ✅ |
| `sync_meta` | `id` (INTEGER) | 单字段主键 | ✅ |
| `sync_changelog` | `id` (INTEGER AUTOINCREMENT) | 单字段主键 | ✅ |
| `backup_records` | `id` (INTEGER AUTOINCREMENT) | 单字段主键 | ✅ |
| `ai_configs` | `id` (TEXT) | 单字段主键 | ✅ |
| `secrets` | `id` (INTEGER AUTOINCREMENT) | 单字段主键 | ✅ |
| `prompts` | `id` (TEXT) | 单字段主键 | ✅ |
| `image_compress_settings` | `minSize` (INTEGER) | 单字段主键 | ✅ |

### 4.1 结论

**所有表均已具备主键，无需修复历史数据，也无需新增 UUID/雪花 ID 字段。**

但需注意以下细节：
- `coupons`、`secrets`、`backup_records`、`sync_changelog` 使用 `INTEGER AUTOINCREMENT`，这在 SQLite 中是本地自增的，不同设备的自增 ID 可能冲突。**建议这些表的新增记录在同步时使用设备级前缀或全局 UUID 来避免冲突。**
- `deal_tags` 和 `deal_promotions` 使用联合主键，在新 changelog 表中需要以组合形式记录主键（如 JSON 数组 `["dealId", "sortOrder"]` 或拼接字符串）。

## 五、可行性评估

### 5.1 总体结论

**新方案整体可行，且相比现有方案在架构完整性上有明显提升。** 但第 3 条要求中"记录 SQL 执行语句"在 Flutter + Drift 技术栈下实现困难，建议调整为"记录变更数据 Payload（JSON）"。

### 5.2 逐项评估

#### 要求 1：所有表需有主键

- **可行性**：✅ 已满足，无需改动。
- **风险**：`AUTOINCREMENT` 表在多设备间存在 ID 冲突风险，建议在同步逻辑中做映射处理。

#### 要求 2：云同步总开关控制 changelog

- **可行性**：✅ 完全可行。
- **实现建议**：
  - 在 `app_settings` 中增加 `cloud.enabled`（已有）和 `cloud.log_changes` 配置。
  - 所有 DAO 的写操作前检查开关状态，或在数据库层通过 `beforeInsert`/`beforeUpdate`/`beforeDelete` 触发器统一控制。
  - 建议封装一个 `ChangeLogger` 中间件，由开关统一管理，而非分散在各 DAO 中判断。

#### 要求 3：新 changelog 表（含 SQL 语句记录）

- **可行性**：⚠️ 部分可行，需调整。
- **问题**：
  - Drift 是 ORM 框架，**不直接暴露最终执行的 SQL 语句**给业务层。虽然可以通过 ` drift_db.logStatements = true` 获取日志，但在生产环境逐条拦截并记录 SQL 会显著影响性能，且 SQL 参数绑定后的语句难以序列化。
  - 即使记录 SQL，跨设备执行同一条 SQL 也可能因自增 ID 差异、时区差异导致不一致。
- **建议调整**：
  - **不记录原始 SQL**，改为记录 **变更后的数据 Payload（JSON）**。
  - 新表结构建议如下：

```sql
CREATE TABLE sync_changelog_v2 (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  device_id TEXT NOT NULL,
  table_name TEXT NOT NULL,           -- 变更表名
  primary_key TEXT NOT NULL,          -- 主键值（联合主键用 JSON 数组）
  operation TEXT NOT NULL,            -- INSERT / UPDATE / DELETE
  changed_at DATETIME NOT NULL,
  synced_at DATETIME,                 -- NULL 表示待同步
  has_attachment INTEGER DEFAULT 0,   -- 是否涉及附件
  attachment_paths TEXT,              -- 附件地址（JSON 数组）
  payload TEXT                        -- 变更数据快照（JSON）
);
```

#### 要求 4：全量上传后清空 changelog

- **可行性**：✅ 完全可行且合理。
- **实现建议**：
  - `SyncService.fullUpload()` 成功后执行 `DELETE FROM sync_changelog_v2 WHERE synced_at IS NOT NULL` 或直接 `DELETE FROM sync_changelog_v2`。
  - 同时重置 `sync_meta.localRevision = 0`，确保后续增量从 1 开始计数。

#### 要求 5：全量下载后弹窗确认

- **可行性**：✅ 完全可行。
- **实现建议**：
  - 全量下载前检查 `sync_changelog_v2` 是否存在 `synced_at IS NULL` 的记录。
  - 若存在，弹窗提供两个选项：
    - **"清空本地变更"**：删除本地 changelog，以远端全量为唯一基线。
    - **"保留并合并"**：保留本地 changelog，在应用全量数据后，继续尝试合并本地变更（需处理冲突）。
  - 注意：若选择"清空"，用户在本地的未同步修改将丢失，需在弹窗中明确提示风险。

#### 要求 6：推送时克隆 changelog 和附件

- **可行性**：✅ 可行，但需明确格式。
- **实现建议**：
  - **不推荐**将整个 SQLite DB 文件作为增量推送（文件大、冲突难处理）。
  - **推荐**将待同步的 changelog 记录序列化为 JSONL 或单个 JSON 文件，连同附件一起打包为 ZIP，推送到远端 `changelog/` 目录。
  - 附件只推送 `has_attachment = 1` 且 `synced_at IS NULL` 对应的文件，避免重复上传。
  - 推送成功后标记 `synced_at`，并定期清理已同步记录。

## 六、新旧方案详细对比

| 对比维度 | 现有方案 | 新方案 | 优劣评估 |
|---------|---------|--------|---------|
| **主键要求** | 已有主键 | 要求所有表有主键 | 已满足，平手 |
| **同步范围** | 仅 `deals` 及其关联表 | 所有业务表 | 新方案胜 |
| **总开关** | 无（依赖 DAO 注入） | 云同步总开关控制 | 新方案胜 |
| **changelog 内容** | entityType/entityId/operation/revision | 表名/主键/方式/payload/附件标记 | 新方案更完整 |
| **SQL 记录** | 无 | 要求记录 | **不可行**，建议改为 payload |
| **附件处理** | 随 deal 一起上传，无独立标记 | 独立标记附件地址，按需推送 | 新方案胜 |
| **全量上传后** | 未清空 changelog | 清空 changelog | 新方案胜 |
| **全量下载后** | 直接覆盖，无提示 | 弹窗让用户选择 | 新方案胜 |
| **增量推送格式** | JSONL 文本文件 | DB/SQL 文件或 JSONL | 建议保留 JSONL |
| **冲突解决** | LWW（Last Write Wins） | 未明确 | 需补充 |
| **多设备并发** | revision 去重 | 需设计 | 需补充 |
| **实现复杂度** | 中等 | 中高 | 现有方案胜 |

## 七、推荐实施路径

若决定实施新方案，建议分阶段推进：

### 阶段 1：基础设施（1-2 周）
1. 创建 `sync_changelog_v2` 表（或扩展现有表）。
2. 实现 `ChangeLogger` 中间件，接入云同步总开关。
3. 在所有业务 DAO（`SettingsDao`、`AiConfigDao`、`SecretsDao`、`PromptDao` 等）的写操作中接入 `ChangeLogger`。

### 阶段 2：增量同步改造（2-3 周）
1. 修改增量推送逻辑：按新表格式序列化，打包附件为 ZIP 推送到 `changelog/`。

### 阶段 3：全量同步优化（1 周）
1. 全量上传成功后清空 changelog 并重置 revision。
2. 全量下载前检测本地 pending changelog，增加弹窗交互。

### 阶段 4：测试与灰度（1-2 周）
1. 多设备并发修改测试。
2. 网络中断恢复测试。
3. 大附件推送性能测试。

## 八、风险与注意事项

1. **AUTOINCREMENT 主键冲突**：`coupons`、`secrets` 等表的自增 ID 在不同设备可能重复，同步时建议用 `deviceId + localId` 组合作为全局唯一键，或改用 UUID。
2. **附件一致性**：图片在本地可能被压缩或替换，需确保 `attachment_paths` 记录的是最终上传时的文件路径。
3. **changelog 表膨胀**：高频操作（如批量导入）会产生大量记录，需设置保留策略（如只保留最近 90 天或 1000 条）。
4. **联合主键序列化**：`deal_tags`（dealId+tag）、`deal_promotions`（dealId+sortOrder）的主键需统一序列化格式（建议 JSON 数组）。
5. **全量下载后合并的复杂性**：若用户选择"保留并合并"，需要处理大量边缘情况（如远端删除了某条 deal，本地又修改了它），建议第一阶段仅支持"清空"，第二阶段再实现智能合并。

## 十、新增要求重新评估

### 10.1 扩展现有 `sync_changelog` 表（要求 1）

**结论：✅ 可行，结构清晰。**

建议的扩展后表结构：

```sql
CREATE TABLE sync_changelog (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  device_id TEXT NOT NULL,
  revision INTEGER NOT NULL,           -- 保留原字段
  table_name TEXT NOT NULL,            -- 新增：变更表名
  primary_key TEXT NOT NULL,           -- 新增：主键值（联合主键用 JSON 数组）
  operation TEXT NOT NULL,             -- 新增：INSERT / UPDATE / DELETE
  changed_at DATETIME NOT NULL,
  synced_at DATETIME,                  -- NULL 表示待同步
  has_attachment INTEGER DEFAULT 0,    -- 新增：是否涉及附件
  attachment_paths TEXT,               -- 新增：附件地址（JSON 数组）
  payload TEXT                         -- 新增：变更数据快照（JSON），替代 SQL
);
```

**注意点**：
- `primary_key` 对联合主键需约定序列化格式，建议统一使用 JSON 数组，如 `["dealId", "sortOrder"]`。
- `payload` 在 DELETE 操作中可以为空或只保留主键信息。
- `revision` 保留后，可与 `device_id` 组合作为去重依据。

---

### 10.2 增量推送格式与 `sync.json` 机制（要求 2、3）

**结论：⚠️ 可行，但存在并发冲突风险，需明确细节。**

#### synclocal.json / synccloud.json 设计

该设计引入了一个**集中式版本协调文件**（`synccloud.json`），相比现有方案的去中心化 JSONL，优势是版本号全局可见，劣势是该文件成为**单点竞争资源**。

```json
{
  "version": 0,
  "lastSyncDeviceId": "123",
  "lastSyncTime": "2023-12-12 12:00:00",
  "changelogZip": [],
  "deviceSyncLog": [
    {
      "deviceId": "123",
      "lastVersion": 0,
      "changelogZip": [],
      "deviceName": "123",
      "lastSyncTime": "2023-12-12 12:00:00"
    }
  ]
}
```

#### 潜在问题分析

| 问题 | 风险等级 | 说明 |
|------|---------|------|
| **并发覆盖** | 🔴 高 | 设备A和B同时基于version=5推送，各自version变为6。后写入的设备会覆盖先写入的`synccloud.json`，先写入设备的`changelogZip`记录丢失，成为孤儿文件。 |
| **version 单调性假设** | 🟡 中 | 方案假设version严格单调递增且全局一致。但并发推送时，两个设备可能同时生成相同的version（都基于5生成6），违反单调性。 |
| **拉取/推送语义混淆** | 🟡 中 | 4.2 中"直接推送增量变更到本地数据库"语义不清，推送应是上传到云端，应用到本地数据库应称为"拉取并应用"。 |
| **changelogZip 清理时序** | 🟡 中 | "超过100条就删除旧的"若由客户端执行，A设备删除后，B设备若尚未同步这些旧包，将永远无法获取。 |
| **网络中断原子性** | 🟡 中 | 上传zip成功但更新`synccloud.json`失败，或反之，均会导致状态不一致。 |

#### 对并发覆盖问题的深入推演

假设设备 A、B 当前本地 version 均为 5，且都有未同步变更：

1. **T1**：A 推送，生成 `changelog_A_6.zip`，将 `synccloud.json` 更新为 version=6，changelogZip 包含 A 的文件。
2. **T2**：B 推送（B 在 T1 前读取了 synccloud.json，version=5，未感知 A 的变更），生成 `changelog_B_6.zip`，将 `synccloud.json` 更新为 version=6，**覆盖**了 A 的 synccloud.json。
3. **结果**：
   - `synccloud.json` 中只有 B 的 changelogZip，A 的 zip 文件成为孤儿。
   - 其他设备 C 下次同步时，version=6，拉取到的 changelogZip 只有 B 的，A 的变更永远丢失。

**解决方向**（需用户确认）：
- 方案 A：**乐观锁 + 重试**。更新 `synccloud.json` 前检查远程 version 是否与本地预期一致，不一致则先拉取再重试。
- 方案 B：**合并而非覆盖**。推送时先拉取远程 `synccloud.json`，将本地新包追加到远程的 `changelogZip` 中，再写回。仍有并发覆盖风险，但可用 ETag/时间戳校验缓解。
- 方案 C：**去中心化**。放弃 `synccloud.json` 的集中版本控制，回归现有方案的 JSONL（每个设备独立文件），通过 `deviceSyncLog` 协调而非全局 version。

同意使用方案A：**乐观锁 + 重试**。更新 `synccloud.json` 前检查远程 version 是否与本地预期一致，不一致则先拉取再重试。

### 10.3 版本比较与同步流向逻辑（要求 4）

**结论：⚠️ 逻辑存在漏洞和语义歧义，需澄清。**

#### 4.1 远程版本 > 本地版本

> "先拉取远程的增量包文件，然后解析增量包文件，应用增量变更到本地数据库。"

**问题**：本地也有未推送的变更（`synced_at IS NULL` 的 changelog 记录），拉取远程后：
- 本地未推送的变更是否还需要继续推送？
- 如果本地和远程修改了同一条记录，冲突如何解决？

**建议流程**：
```
拉取远程增量包 → 应用到本地 → 合并冲突（LWW） → 标记本地已同步到该版本
→ 若本地仍有未同步变更 → 继续推送（生成新的 version）
```

#### 4.2 远程版本 < 本地版本

> "直接推送增量变更到本地数据库。"

**问题**："推送"与"本地数据库"语义矛盾。推送是上传到云端，应用到本地应叫"拉取"。

**推测用户意图**：远程版本更低，说明本地有更新，应直接**将本地增量包上传到云端**。

#### 4.3 远程版本 = 本地版本

> "判断上次同步设备id是否是本设备，如果是则直接推送...如果不是本设备，则需要先拉取远程..."

**问题 1**：version 相等意味着自上次同步以来没有设备成功完成过完整的推送+版本更新。此时：
- 如果是本设备：说明上次是本地推送的，本地可能有新变更也可能没有。如果没有新变更，推送什么？
- 如果不是本设备：说明上次是其他设备推送的，但 version 和本地一样，意味着远程没有新变更，拉取什么？

**问题 2**：该逻辑实际上等价于**无论 lastSyncDeviceId 是什么，只要本地有未同步变更就应该推送**。version 相等时远程没有新内容，无需拉取。

**建议简化**：
```
若 远程 version > 本地 version：拉取远程增量 → 应用 → 若本地仍有未同步 → 推送
若 远程 version <= 本地 version：直接推送本地未同步增量（如果有）
```

`lastSyncDeviceId` 可作为日志追溯用途，不建议作为同步流向的决定因素。

## 十二、修订后的推荐流程

基于以上分析，若用户接受以下调整，方案可行性将大幅提升：

### 调整 1：解决并发冲突
- 放弃 `synccloud.json` 的**覆盖式更新**。
- 改为**读取远程 → 合并 changelogZip → 写回**，配合 ETag/时间戳检测冲突，冲突时重试。

### 调整 2：简化版本比较逻辑
```
1. 读取远程 synccloud.json
2. 若远程 version > 本地 version：
   a. 拉取远程增量包并应用到本地
   b. 更新本地 version = 远程 version
   c. 若本地仍有未同步变更 → 继续步骤 3
3. 若本地有待同步变更：
   a. 打包本地增量 + 附件为 ZIP
   b. 上传 ZIP 到云端
   c. 读取远程 synccloud.json（可能已有其他设备更新）
   d. 合并 changelogZip，version + 1，更新 lastSyncDeviceId
   e. 写回 synccloud.json（带冲突检测）
   f. 标记本地 changelog 为已同步
   g. 更新 synclocal.json
```

### 调整 3：全量同步后的 changelog 管理

**全量上传成功后**：
1. 清空本地 `sync_changelog` 表中所有记录（或仅清空已同步的）。
2. 保留 `synclocal.json` 的 `version` 值（不重置为 0），仅清空 `changelogZip` 数组。
3. 上传新的全量备份到 `full/` 目录。

**全量下载时**：
1. 检查本地 `sync_changelog` 是否存在 `synced_at IS NULL` 的记录。
2. 若存在，弹窗提示："本地有未同步的变更，继续下载将丢失这些变更，是否继续？"
3. 用户确认后，执行现有替换逻辑：关闭数据库 → 下载 ZIP → 替换文件 → 重启应用。
4. 替换完成后，若本地无 `synclocal.json`，则初始化（`version=0`, `lastSyncDeviceId=本设备`）。

### 调整 4：synclocal.json 初始化规则

- **首次开启云同步总开关**：仅建立云存储连接，不生成 `synclocal.json`。
- **首次全量上传/下载完成后**：生成本地 `synclocal.json`：
  ```json
  {
    "version": 0,
    "lastSyncDeviceId": "{本设备ID}",
    "changelogZip": [],
    "deviceSyncLog": [
      {
        "deviceId": "{本设备ID}",
        "lastVersion": 0,
        "changelogZip": [],
        "deviceName": "{设备名称}"
      }
    ]
  }
  ```
