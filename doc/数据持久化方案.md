# 折多多 · 数据持久化方案

> **选型结论：SQLite + 外置图片目录 + WebDAV 增量云同步**  
> 当前原型仍用 `localStorage` 演示；正式版按本文实施。

---

## 一、方案概述

| 维度 | 方案 |
|------|------|
| 本地存储 | SQLite 3（`zheduoduo.db`） |
| 图片 | 外置 `images/{deal_id}.jpg`，库内只存路径与元数据 |
| 网络模式 | **离线优先**，无网完整可用 |
| 云同步 | WebDAV 增量 changelog + 图片文件；COS/OSS 同协议 |
| 备份 | zip（db + images + manifest） |
| 导入导出 | zip 恢复；另支持 JSON/YAML 单条导入 |

详细表结构见 **[数据库表设计.md](./数据库表设计.md)**。

## 三、离线优先

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  UI / Vue   │ ──▶ │ Repository   │ ──▶ │ SQLite      │
└─────────────┘     └──────────────┘     └─────────────┘
                           │
                           ▼
                    ┌──────────────┐
                    │ images/      │
                    └──────────────┘
                           │
              （有网且 autoSync）▼
                    ┌──────────────┐
                    │ WebDAV Push  │
                    └──────────────┘
```

- 所有 CRUD 只写本地，响应即时
- 变更写入 `sync_changelog`，标记 `synced_at IS NULL`
- 网络恢复后后台上传，失败重试不阻塞用户

## 四、云同步（WebDAV）

### 远端结构

```
remote/zheduoduo/
├── manifest.json                              # 全局版本与文件清单
├── db/
│   ├── changelog_{deviceId_1}.jsonl           # 设备 1 的增量变更
│   ├── changelog_{deviceId_2}.jsonl           # 设备 2 的增量变更
│   └── ...
├── full/                                      # 当前全量（全量 Push 覆盖）
│   ├── zheduoduo.db                           # 最新完整数据库
│   └── manifest.json                          # 全量元信息
├── snapshots/                                 # 历史快照（只增不改）
│   └── zheduoduo-YYYYMMDD.db.zip
└── images/{deal_id}.jpg
```

### 同步策略

同步分为**增量**（日常自动）和**全量**（手动触发）两种模式：

| 操作 | 模式 | 触发方式 | 说明 |
|------|------|----------|------|
| Push 增量 | 自动 | 保存后 debounce 3s | 上传变更图片 + changelog 行 |
| Pull 增量 | 自动 | 前台恢复时 | 下载 manifest → 拉取新 changelog → LWW 合并 → 下载缺失图片 |
| **Push 全量** | **手动** | 用户点击"全量上传" | 上传完整 db + 所有图片，覆盖远端，清空 changelog |
| **Download 全量** | **手动** | 用户点击"全量下载" | 下载 full/ 替换本地 db + 图片，重置 revision |
| 冲突 | — | — | `updated_at` 较新者胜；删除进回收站，需对方确认 |

> 全量操作涉及完整数据库和所有图片，文件体积可能较大，**仅支持手动触发**，不设自动。

#### 全量 Push（上传整库 → 覆盖云端）

适用场景：首次设置云端、换设备后以当前设备为准、手动导入 zip 后同步。

```
用户点击"全量上传"
        │
        ▼
  弹窗确认："将覆盖云端所有数据，是否继续？"
        │ 确认
        ▼
  ① 本地 VACUUM INTO 压缩数据库 → 临时文件
  ② 上传压缩 db → full/zheduoduo.db
  ③ 上传本地 images/*.jpg → images/（跳过已存在且大小一致的）
  ④ 写入 full/manifest.json（时间、设备、记录数、图片数）
  ⑤ 更新远端根 manifest.json（remoteRevision、devices、full 字段）
  ⑥ 清空远端所有 changelog_{deviceId}.jsonl（已合并进全量）
  ⑦ 本地 sync_meta 重置：remote_revision = local_revision
  ⑧（可选）在 snapshots/ 保留一份快照归档
```

#### 全量 Download（下载整库 → 覆盖本地）

适用场景：新设备首次登录、本地数据库损坏、想以云端为准。

```
用户点击"全量下载"
        │
        ▼
  检查本地是否有未推送的 changelog
        │
    ┌───┴───┐
    │ 有     │ 无
    ▼       ▼
  二次确认：  直接继续
  "本地有 N 条
  未同步修改，
  将被丢弃"
        │ 确认
        ▼
  ① 下载 full/zheduoduo.db → 替换本地 zheduoduo.db
  ② 下载 full/manifest.json → 读取图片清单
  ③ 对比本地 images/，下载缺失/不同的图片
  ④ 删除本地多余图片（云端没有的）
  ⑤ 重置本地 sync_meta：local_revision = remote_revision
  ⑥ 清空本地 sync_changelog（已对齐全量）
```

#### 删除冲突策略：回收站机制

删除操作**不直接写 `deleted=1`**，而是先进"回收站"：

1. 本地删除时：`deleted` 设为 `pending_delete`（而非 `1`），记录 `deleted_at`
2. Push 时：`pending_delete` 状态随 changelog 上传
3. 对方 Pull 时：发现 `pending_delete` 记录 → 弹窗确认
   - 确认删除 → `deleted = 1`
   - 取消删除 → `deleted = 0`，保留修改内容
4. 确认结果作为新变更写入对方 changelog，下次 push 同步回删除方

> 此机制避免"一边编辑一边删除导致用户数据静默丢失"的问题。

#### changelog 按设备分文件

每台设备 push 到独立文件 `changelog_{deviceId}.jsonl`，而非追加到同一文件：

- **天然避免并发覆盖**：不同设备写不同文件，无竞态
- **pull 时合并**：下载所有设备的 changelog 文件，按 revision 排序后逐条应用
- **清理策略**：当所有设备的 `local_revision` 均 ≥ 某文件的最大 revision 后，该文件可归档到 snapshots

#### 图片上传顺序

Push 时**先上传图片，再写 changelog**：

1. 上传 `images/{deal_id}.jpg`
2. 上传成功后，将 changelog 行写入远端 `changelog_{deviceId}.jsonl`
3. 更新本地 `synced_at`

> 保证对方 pull 到 deal 数据时，图片已在远端。若图片上传失败，changelog 不写入，下次重试。

#### Pull 触发机制

- 保存 Deal 后 debounce 3s 触发 push
- **App 前台恢复时立即触发 pull**（监听 `visibilitychange` / `appStateChange`）
- 可选定时 pull（如 30min），但以前台恢复为主触发源

设置项（存 `app_settings`）：

- `cloud.webdav.enabled` — 启用云同步
- `cloud.webdav.autoSync` — 保存后自动 push / 前台恢复时 pull
- `cloud.webdav.lastSyncAt` — 上次同步时间

## 五、图片规范

| 步骤 | 规则 |
|------|------|
| 来源 | 本地上传 / YAML `image_url` 下载 |
| 压缩 | 宽 ≤800px，JPEG quality 0.7 |
| 存储 | `images/{dealId}.jpg` |
| 数据库 | `deal_images.image_path`，禁止存 base64 |
| 删除 | 软删 Deal 时保留图片至同步完成，硬删时移除文件 |

## 七、JSONL 的定位

JSONL 不再作为主存储，仅作：

- YAML/AI 批量导入的中间格式
- 用户「导出为 JSONL」的可读备份
- 调试与数据交换

---

## 八、为何不用 JSONL 主存

| 维度 | JSONL | SQLite（选用） |
|------|-------|----------------|
| 多条件筛选 | 弱 | SQL + 索引 |
| 事务 / 一致性 | 弱 | 强 |
| 单条更新 | 重写文件 | UPDATE |
| 离线 + 增量同步 | 可行但需自建索引 | changelog 表原生支持 |
| 数据量 | <5000 尚可 | 万级以上稳定 |
